# Time-Series Ingestion Bandwidth Optimization PRD

## Status

Implemented for the `0.1.3` release path in commit `ebe65b2`.

Validated against `wearables-demo` dev deployment `dev:fortunate-chickadee-603` on `2026-04-27`.

Implementation covered the bandwidth-critical path:

- removed inline maintenance from `storeBatch`
- made raw point upserts batch-aware
- made rollup upserts batch-aware
- increased Garmin webhook data point batch size from `100` to `500`
- made scheduled maintenance bounded and backlog-aware
- added tests for mixed upsert replay and maintenance backlog behavior

Deferred from the broader proposal:

- Garmin daily payload idempotency table
- explicit ingestion telemetry counters
- deeper no-op rollup maintenance scan optimization beyond the bounded maintenance fix

## Problem

Convex account usage showed unexpectedly high database bandwidth. The dominant source was not Estulio embeddings. In the account breakdown by function, the largest database bandwidth consumer was the `test` dev deployment:

- `test (test-79c75)`
- `wearables/dataPoints.storeBatch`
- Approximately `87.71 GB` database bandwidth in the selected billing window

Other relevant usage from the same screenshot:

- `clipin-app`: `wearables/dataPoints.storeBatch` around `7.18 GB`
- `estulio`: total database bandwidth around `2.28 GB`

This means the primary optimization target is the `convex-wearables` component time-series ingestion path.

The storage policy is already useful for limiting retained row count. The issue is different: database bandwidth is generated by repeated reads and writes while receiving and processing dense data, even if long-term storage remains capped around 9k to 9.5k rows per user.

5. Repeated maintenance reads dominate database bandwidth.

## Current Implementation Path

### Garmin daily expansion

File:

- `src/component/providers/garmin.ts`

`normalizeDaily` expands `daily.timeOffsetHeartRateSamples` into absolute timestamp/value pairs:

```ts
heartRateSamples = Object.entries(daily.timeOffsetHeartRateSamples).map(
  ([offsetStr, bpm]) => ({
    timestamp: baseMs + Number(offsetStr) * 1000,
    value: bpm,
  }),
);
```

For dense Garmin heart rate at around 15-second cadence, one day can contain thousands of samples. The existing storage policy documentation estimates:

- `4` rows per minute
- `240` rows per hour
- `5760` rows per day

For a `dailies: 22` webhook, tens of thousands of normalized points are realistic.

### Webhook batching

File:

- `src/component/garminWebhooks.ts`

Current constant:

```ts
const DATA_POINT_BATCH_SIZE = 100;
```

`storeNormalizedDataPoints` groups by `seriesType`, then sends each group to:

```ts
internal.dataPoints.storeBatch
```

in batches of 100 points.

The log sample produced `932` `storeBatch` calls, which is consistent with roughly 93k normalized data points for one webhook.

### Hot mutation

File:

- `src/component/dataPoints.ts`

`storeBatch` calls `storePointsWithPolicy`.

`storePointsWithPolicy` currently:

1. Reads the `dataSources` document.
2. Resolves the effective policy.
3. Reads time-series policy settings.
4. Deduplicates the incoming batch only in memory.
5. Splits points into raw points or rollup groups based on age and policy.
6. Upserts raw points or rollup buckets.
7. Upserts `timeSeriesSeriesState`.
8. If policy requires maintenance, runs `maintainSeriesState` inline.
9. Schedules maintenance.

The most expensive line of behavior is the inline maintenance call:

```ts
if (policyRequiresMaintenance(effectivePolicy.tiers)) {
  await maintainSeriesState(...);
  await ensureTimeSeriesMaintenanceScheduled(ctx);
}
```

### Repeated rollup scan

File:

- `src/component/dataPoints.ts`

`maintainSeriesState` calls:

```ts
await compactRawPoints(...);
await compactRollupPoints(...);
```

`compactRollupPoints` currently reads up to 500 rollup rows:

```ts
const rollups = await db
  .query("timeSeriesRollups")
  .withIndex("by_source_type_bucket", (idx) =>
    idx.eq("dataSourceId", state.dataSourceId).eq("seriesType", state.seriesType),
  )
  .take(MAINTENANCE_ROLLUP_BATCH_SIZE);
```

This scan happens inside every `storeBatch` call for tiered policies. For the observed webhook:

- `932` `storeBatch` calls
- each reading roughly `543` to `555` documents
- approximately `503k` total document reads

This strongly indicates the same or similar rollup/state rows are being read repeatedly during one webhook.

## Current Policy In Demo

`../wearables-demo/convex/wearablesPolicy.ts` defines a 30-day retention policy:

- default unspecified metrics: raw for 30 days, then deleted
- `heart_rate`: raw for 48h, 15m rollups until 14d, 1h rollups until 30d
- `oxygen_saturation`: raw for 24h, 30m rollups until 30d
- `respiratory_rate`: raw for 24h, 30m rollups until 30d
- Garmin stress/body battery: raw for 24h, 30m rollups until 30d
- maintenance interval: 1h

Verified from `test` dev deployment:

```bash
cd ../wearables-demo

```

The policy is helping storage, but it does not prevent high ingest-time database bandwidth because the hot mutation still repeatedly performs point upserts and maintenance scans.

## Proposed Changes

### Change 1: Remove Inline Maintenance From `storeBatch`

The most important change is to stop doing full maintenance inside the hot ingest mutation.

Current behavior:

- Every tiered `storeBatch` call may run `maintainSeriesState`.
- During dense webhooks, this repeats hundreds of times.

Target behavior:

- `storeBatch` stores/updates the incoming batch.
- `storeBatch` updates series state.
- `storeBatch` marks the series as due for maintenance.
- `storeBatch` ensures a scheduled maintenance job exists.
- Full compaction happens in `runTimeSeriesMaintenance`, not inline.

Expected effect:

- Dense webhook processing should no longer read 500 rollups per 100-point batch.
- `storeBatch` read documents should become proportional to the incoming batch and actual duplicate/rollup upserts, not to total existing rollup count.

Functional impact:

- Public behavior should remain the same.
- Old raw points may remain raw until the scheduled maintenance job runs.
- Since reads are policy-aware and have fallback paths, this should not break user-facing charts if implemented carefully.

Implementation notes:

- Add or update a helper that marks a series state due without running maintenance immediately.
- Keep `ensureTimeSeriesMaintenanceScheduled`.
- Avoid patching policy settings on every batch if a future maintenance run is already scheduled.
- Preserve the `policyRequiresMaintenance` check, but use it to decide scheduling/state, not inline compaction.

Acceptance criteria:

- For dense webhook ingestion, `dataPoints:storeBatch` should no longer read around 500 documents per call.
- `runTimeSeriesMaintenance` should be the main function that performs compaction reads.
- Existing tests for storage policy and time-series reads must pass.

### Change 2: Make Raw Point Upserts Batch-Aware

Current behavior:

- `upsertRawPoints` performs one indexed query per point:

```ts
for (const point of points) {
  const existing = await db
    .query("dataPoints")
    .withIndex("by_source_type_time", ...)
    .first();
  ...
}
```

Target behavior:

- For each `storeBatch`, determine `min(recordedAt)` and `max(recordedAt)`.
- Query existing raw points for that `dataSourceId`, `seriesType`, and timestamp range once, or in bounded chunks if needed.
- Build an in-memory map keyed by `recordedAt`.
- Insert missing points.
- Patch only changed points.
- Skip unchanged points.

Expected effect:

- Duplicate or replayed Garmin payloads become much cheaper.
- Backfills that overlap previous data avoid unnecessary writes.
- Read count drops from one existence query per point to one bounded range query per batch.

Functional impact:

- Same logical upsert behavior.
- Existing points still get updated if value or `externalId` changes.
- Exact duplicate points do not generate patches.

Acceptance criteria:

- Replaying the same raw point batch does not patch unchanged rows.
- Replaying the same raw point batch has low write bytes.
- Unit tests cover insert, changed update, unchanged skip, and mixed batches.

### Change 3: Batch-Aware Rollup Upserts

Current behavior:

- `upsertRollupStatsGroups` checks each rollup group individually.

Target behavior:

- Group rollup updates by `bucketMs`.
- Query existing rollups for each affected `bucketMs` and bucket range.
- Build an in-memory map keyed by `bucketMs:bucketStart`.
- Insert missing rollups.
- Patch changed rollups.
- Skip unchanged rollups when possible.

Expected effect:

- Old dense samples that route directly to rollups require fewer reads and writes.
- Replayed backfill data becomes cheaper.

Important correctness detail:

- Combining rollup stats must remain mathematically correct.
- The current `combineAggregatedStats` behavior must be preserved unless a separate correctness issue is discovered and documented.

Acceptance criteria:

- Existing rollup behavior tests still pass.
- New tests cover rollup insert, update, and replay.

### Change 4: Increase Webhook Data Point Batch Size Carefully

Current Garmin webhook batch size:

```ts
const DATA_POINT_BATCH_SIZE = 100;
```

Target:

- Increase this after removing inline maintenance and improving batch upserts.
- Start with `500` or `1000`.
- Validate against Convex mutation limits using tests and the `test` dev deployment.

Expected effect:

- Fewer `storeBatch` calls per webhook.
- Fewer repeated policy/settings/state reads.
- Less function overhead.

Risk:

- Too-large batches may hit Convex mutation read/write/document/transaction limits.

Acceptance criteria:

- A `dailies` payload similar to the observed 22-day webhook should produce far fewer `storeBatch` calls than 932.
- No transaction limit errors in tests or dev logs.

### Change 5: Add Garmin Daily Idempotency

Dense Garmin daily payloads can be replayed by webhook retries, backfill retries, or repeated manual testing.

Target behavior:

- Before expanding a Garmin daily into thousands of samples, detect whether the exact daily payload has already been processed.
- If exact same daily was already processed, skip expansion and storage.
- If the same daily id arrives with changed contents, process it.

Possible implementation:

- Add a small component table, for example `providerPayloadIngestions` or `garminDailyIngestions`.
- Key fields:
  - `provider`
  - `providerUserId`
  - `dataType`
  - `externalId` or `summaryId`
  - `payloadHash`
  - `processedAt`
  - optional counts for observability
- Index by provider/user/data type/external id.

Hashing requirement:

- Compute a stable hash of the normalized relevant payload in the action.
- Do not store raw Garmin payloads in Convex just for idempotency.

Functional impact:

- Exact duplicate dailies are skipped.
- Updated Garmin daily records are still processed.

Acceptance criteria:

- Processing the same daily payload twice does not call `storeBatch` the second time.
- Processing a daily with same id but different samples does process updates.

### Change 6: Improve Scheduled Maintenance Scans

Even after removing inline maintenance, scheduled maintenance should avoid repeatedly reading the same non-actionable rollup rows.

Current issue:

- `compactRollupPoints` reads the first 500 rollups for a source/series.
- Rows already in their correct final bucket tier may be read again and again.

Target behavior:

- Query only rollups that can actually move to a coarser tier or expire.
- Prefer the `by_source_type_bucket_size` index when bucket size matters.
- For tier transitions, query source bucket sizes that are smaller than the destination bucket size.
- Avoid scanning final-tier rollups on every run.

Potential schema extension:

- Add maintenance cursor fields to `timeSeriesSeriesState` if needed.
- Use cursors only if index targeting is not enough.

Acceptance criteria:

- `runTimeSeriesMaintenance` read volume is proportional to rows that can be compacted/deleted, not total retained rollup rows.
- Repeated maintenance runs with no eligible rows should be cheap.

### Change 7: Add Ingestion Telemetry

Convex function-level usage is useful but not enough to explain payload composition or ingest decisions.

Add lightweight logs or small telemetry counters for development and debugging:

- payload type
- input point count
- raw point count
- direct rollup group count
- inserted raw rows
- patched raw rows
- skipped unchanged raw rows
- inserted rollups
- patched rollups
- skipped duplicate dailies
- maintenance scheduled or not

Prefer structured `console.info` summaries over logging raw payload data.

Acceptance criteria:

- Logs can explain why a webhook was expensive without exposing raw health data.
- Raw Garmin payloads are not printed.

### 9. Compare Before vs After

Baseline from the observed problematic webhook:

```text
payload: dailies: 22
storeBatch executions: 932
storeBatch read MB: 188.467
storeBatch write MB: 2.041
storeBatch read docs: 503,638
HTTP duration: 151.866s
```

Target after optimization for a similar payload:

```text
storeBatch executions: at least 5x lower, preferably 10x lower
storeBatch read MB: at least 90% lower, target under 20 MB
storeBatch read docs: at least 90% lower, target under 50,000
HTTP duration: under 1s if async webhook scheduling is active
processing action duration: lower than baseline and no timeout risk
```

The most important signal:

- Individual `dataPoints:storeBatch` calls should not regularly read about 500 documents.

If scheduled maintenance now performs compaction, that is acceptable, but it should happen under:

- `wearables/dataPoints:runTimeSeriesMaintenance`

not repeatedly inside:

- `wearables/dataPoints:storeBatch`

## Risks And Mitigations

### Risk: Delayed compaction changes read results

Mitigation:

- Keep `getTimeSeries` policy-aware.
- Preserve fallback reads between raw and rollup tiers.
- Add tests that query immediately after ingest and after maintenance.

### Risk: Larger batches hit mutation limits

Mitigation:

- Increase gradually.
- Start with `500` before `1000`.
- Validate in Convex logs.
- Keep batch size constants easy to tune.

### Risk: Idempotency skips legitimate updates

Mitigation:

- Skip only exact same payload hash.
- Reprocess same external id if hash changed.

### Risk: Scheduled maintenance backlog grows

Mitigation:

- Mark affected series due.
- Keep scheduled maintenance coalesced.
- Optimize maintenance scans so no-op runs are cheap.

### Risk: npm/local package mismatch during demo testing

Mitigation:

- Use `npm ls @clipin/convex-wearables`.
- Restart `npx convex dev`.
- Confirm logs show `Garmin webhook ingestion scheduled` if local source includes async scheduling.
